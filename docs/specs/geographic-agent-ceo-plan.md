## Vision

### 10x Check
A live geographic demand map that updates monthly. Each month, the walk-forward loop
validates last month's predictions against reality. The system learns which categories
have predictable growth patterns and which states have underpriced seller supply.
After the walk-forward backtest (**12** completed iterations; prediction months **Oct 2017–Aug 2018**), Olist has a ranked opportunity pipeline with
confidence scores — not a gut feeling.

> The 10x version adds real-time seller onboarding recommendations, automated outreach
> triggers, and a category expansion risk score. NOT in MVP scope.

## Business Problem

Predict which of the top 5 categories grows in which of the top 5 states next month.
Identify the seller supply gap. Rank opportunities for Olist to act on.

**Why this matters:** Olist has 0% repeat customer rate. Growing into new geographic
demand without corresponding seller supply just delays bad reviews. The gap between
predicted demand and actual seller capacity is where Olist loses money.

## v2 MVP Scope

### 1. What Is Built

#### A. GeographicAgent (new agent)

`agents/geographic/geographic_agent.py`

Consumes: domain agent outputs (customer, product, seller)
Produces: structured geographic momentum + supply gap analysis

Does NOT subclass BaseAgent (same pattern as ConnectorAgent — different interface).

Key methods:
- `_load_geographic_data()` — joins per engineering join chain; load **categories** CSV with **`encoding='utf-8-sig'`** (BOM); **assert** null English category **< 1%** after join
- `_compute_growth_matrix(training_df)` — MoM growth per (`customer_state`, category): **period continuity** check **before** `pct_change()`; **reindex** missing months with **0**, mark affected steps **LOW** confidence; **after** `pct_change()`: **`replace([inf, -inf], nan)`**; **zero-order** months → **NaN** growth, **LOW** confidence; **month-level** guard: if **total orders in that month < 10** → **NaN** growth, **LOW** confidence (see Key Design Decisions)
- `_identify_top5_states()` — by total revenue in training window (`customer_state`)
- `_identify_top5_categories()` — by total revenue in training window
- `_predict_next_month_growth(state, category, momentum)` — LLM-assisted prediction; outputs **`confidence_score`** alongside **`confidence`**
- `_compute_supply_gap(...)` — supply from **`seller_state`**; demand/order counts from **`customer_state`** (see Key Design Decisions)
- `run() -> GeographicOutput`

#### B. walk_forward.py (new orchestrator)

The core architecture. Data spans **Sep 2016 through Oct 2018** (**25** distinct calendar months in the raw extract). With a **12-month** rolling training window, there are **13** possible prediction months (25 − 12). **Completed** walk-forward runs **12** iterations: prediction months **Oct 2017 through Aug 2018**. The **13th** slot would predict **Sep 2018**, which has **1 order** total in the dataset — **exclude** it from the completed run and from validation (engineering review).

Do **not** describe the loop as “25 iterations” or “months 13–37”.

**Month indexing**

| Role | Calendar |
|------|-----------|
| First month in data | Sep 2016 |
| Last month in data | Oct 2018 |

**Iteration → calendar mapping (rolling 12m train; aligned to actual Olist months)**

- **First prediction month:** **Oct 2017** (first month after a full year from dataset start; not “Jan 2017” or “month 13 = Jan 2017”).  
- **Completed predictions:** **Oct 2017 – Aug 2018** (**12** iterations).  
- **Excluded:** prediction target **Sep 2018** (degenerate volume — 1 order total).  
- **Maximum** train→predict slots from calendar math: **13** (25 months − 12); MVP completes **12** (not 25).

```
months = sorted(df['month'].unique())   # 25 calendar months
usable_months = [m for m in months if monthly_order_count[m] >= MIN_ORDERS]  # exclude sparse ends as needed
# With 12-month window: at most len(usable_months) - 12 prediction steps.
# MVP: completed_iterations == 12; do not assert 25.

for iteration i from 1 to 12:   # completed run (predict Oct 2017 .. Aug 2018)
    training_window = 12 calendar months ending the month before prediction_month(i)
    prediction_month = month i of the agreed schedule  # see integration tests / fixture table

    geographic_agent.compute_growth(training_window)
    geographic_agent.predict(prediction_month)
    opportunity_ranker.rank_opportunities()   # composite uses confidence_score (see Schema)
    store_prediction(prediction_month, results)

    if i > 1 and actuals_available(prediction_month):
        score_previous_prediction()
```

**Reference mapping (engineering review; training spans may have 11 distinct months if Nov 2016 is missing — see `temporal_window` month_count assert):**

| i | Prediction month (completed run) |
|---|----------------------------------|
| 1 | 2017-10 |
| 2 | 2017-11 |
| … | … |
| 12 | 2018-08 |

*(Iteration that would target **2018-09** is out of MVP completion criteria.)*

Output per iteration is a **`WalkForwardIteration`** (`utils/schema_geographic.py`): includes **`training_window`**: `tuple[str, str]` **(start_month, end_month)**, `prediction_month`, **`previous_prediction`** (prior iteration’s `predictions`, or `None` on iteration 1), `predictions`, `supply_gaps` (with `supply_gap_ratio` / sparse-cell guard per above), `ranked_opportunities`, and `validation`. See the **Schema Extensions** TypedDict block below for the full shape.

**Sparse-cell guard:** If `current_month_order_count < 10` for a (state, category) cell, set **confidence = LOW** and **skip** demand/supply gap math for that cell (omit or null `predicted_order_volume` and `supply_gap_ratio`).

#### C. OpportunityRanker

Ranks opportunities by: **`predicted_growth × confidence_score × supply_gap_severity`** (cells with skipped demand are excluded or deprioritized). Use numeric **`confidence_score`** from **`Prediction`** — not the string **`confidence`** label (avoids `TypeError` at runtime).

**Supply-side ratio (replaces `gap_units`):**

- `predicted_order_volume = current_month_order_count × (1 + predicted_growth)`  
  (`predicted_growth` as a fraction, consistent with MoM convention in implementation.)
- `supply_gap_ratio = predicted_order_volume / max(current_sellers, 1)`  
- **Guard:** if `current_month_order_count < 10`, set **confidence = LOW** and **skip** demand calculation (no `predicted_order_volume` / `supply_gap_ratio` for that cell).

Supply gap severity is derived from `supply_gap_ratio` (higher = more pressure per seller).

Ranked output:

Include these TypedDicts: **GeographicMetrics**, **Prediction**, **SupplyGap**, **RankedOpportunity**, **GeographicOutput**, **WalkForwardIteration**, **WalkForwardResult**.

```python
    top5_states: list[str]                          # state codes e.g. ["RJ", "SP", ...]
    top5_categories: list[str]                     # category names
```

## Key Design Decisions

**Q: LLM for prediction or pandas trend extrapolation?**
A: Hybrid. Pandas computes momentum (slope of last 3 MoM growth rates). LLM consumes momentum + growth matrix + supply gap data and produces a structured prediction with confidence. This follows the v1 rule: pandas computes facts, LLM synthesizes narrative.

**Q: Which `state` for seller supply vs demand?**
A: **`seller_state`** (sellers table) defines **supply-side** seller counts for the gap — count sellers **physically in** that state (`seller_id.nunique()` where **`seller_state == focus_state`**). **Demand** (revenue, MoM growth, order volumes) uses **`customer_state`** from the customers join. Do **not** use **`customer_state`** on the seller join for supply; that conflates “sellers who shipped to RJ” with “sellers in RJ.”

**Join chain (reference) for `_load_geographic_data()`:**  
orders → customers (**`customer_state`**) → order_items → products → categories → sellers (**`seller_state`**).

**Q: Why fixed top-5 at iteration 1?**
A: The problem is "which of the top-5 categories grows in which of the top-5 states." If we re-compute top-5 each iteration, we're answering a different question each time. Fixed top-5 from the initial training window gives us a consistent scope across all **12** completed iterations for fair backtesting.

**Q: Confidence scoring?**
A: Three-level (HIGH/MEDIUM/LOW) **and** numeric **`confidence_score`** (e.g. HIGH=1.0, MEDIUM=0.6, LOW=0.3) for ranking math. HIGH = 3+ months of consistent momentum direction. MEDIUM = 2 months. LOW = 1 month, conflicting signals, sparse cells, bad growth series, or **<10 orders** for that **(state, category)** cell. **Composite score** uses **`confidence_score`**, not the string **`confidence`**.

**Q: Months with very low total order volume?**
A: If **total orders in that calendar month** (dataset-level or slice used for the growth step — as implemented) is **< 10**, treat growth as **NaN** and set **confidence = LOW** / **`confidence_score`** at the low tier. (Aligns with engineering review Option B.)

**Q: Supply gap ratio?**
A: **`predicted_order_volume = current_month_order_count × (1 + predicted_growth)`** and **`supply_gap_ratio = predicted_order_volume / current_sellers`** (use `max(sellers, 1)` if needed). This replaces `gap_units`. If **`order_count < 10`** for a **(state, category)** cell, set confidence **LOW** and **skip** demand calculation for that cell (no ratio).

**Q: `pct_change()` and bad floats?**
A: After **`pct_change()`**, **`replace([np.inf, -np.inf], np.nan)`**. **Zero-order** months yield **NaN** growth and **LOW** confidence — never propagate **`inf`** into momentum or the LLM. **Before** `pct_change()`, enforce **period continuity**; **reindex** missing periods with **0** and treat adjacent steps as **LOW** confidence (e.g. Nov 2016 missing).

**Q: Why not use ML forecasting (ARIMA, Prophet)?**
A: 12 months of monthly data is too sparse for classical ML. LLM with momentum scoring is interpretable and can incorporate business context (seasonality, product launches) that ARIMA can't. This is an AI-assisted forecast, not a statistical forecast.

**Q: Temporal coherence — how to avoid T+1 predicting the opposite of T?**
A: Each iteration's LLM prompt includes the previous iteration's prediction. This makes the LLM aware of trend continuity. **`WalkForwardIteration.previous_prediction`** carries the prior iteration's prediction grid for both JSON output and prompt context. This costs nothing and makes predictions temporally coherent.

**Q: `temporal_window()` / training bounds?**
A: Returns **`tuple[DataFrame, int]`** so the caller can **assert** the **distinct month count** matches the intended window (avoids silently using 11 months when Nov 2016 is missing). **`GeographicOutput`** / **`WalkForwardIteration`** expose **`training_window: tuple[str, str]`** **`(start_month, end_month)`** for debugging.
