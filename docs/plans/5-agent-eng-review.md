### Finding 1 — [P1] (confidence: 9/10) Payment Join Row Explosion NOT Fully Fixed

**PRIOR LEARNING APPLIED:** `payment-join-row-explosion` (confidence: 9/10, 2026-04-18)

The CEO plan notes the payment row explosion was "fixed," but the fix is incomplete. The join chain for `CustomerReadyAgent` still merges raw `payments`:

```python
# SPEC AS WRITTEN — WRONG
orders
  .merge(customers, on="customer_id", how="inner")
  .merge(order_items, on="order_id", how="inner")
  .merge(products, on="product_id", how="left")
  .merge(categories, on="product_category_name", how="left")
  .merge(payments, on="order_id", how="left")   # ← raw payments → row explosion
```

Olist's `payments` table has one row per `payment_sequential` per order. A 6-installment credit card order = 6 rows. Joining raw `payments` multiplies ALL rows by the installment count, corrupting:
- `repeat_rate` — one order appears as 6 "orders" per customer
- `order_volume_trend` — MoM counts inflated by avg installment count
- `high_value_customer_count` — avg payment computed per installment, not per order

The plan's computation step ("aggregate payments to order level first for avg_spend") only fixes `avg_spend`. The other 3 metrics remain broken.

**Fix (accepted by user):** Pre-aggregate payments before the merge:

```python
payments_agg = (
    payments
    .groupby("order_id")["payment_value"]
    .sum()
    .reset_index()
    .rename(columns={"payment_value": "total_payment_value"})
)

orders
  .merge(customers, on="customer_id", how="inner")
  .merge(order_items, on="order_id", how="inner")
  .merge(products, on="product_id", how="left")
  .merge(categories, on="product_category_name", how="left")
  .merge(payments_agg, on="order_id", how="left")   # aggregated — safe
```

`avg_spend` then = `mean(total_payment_value)` per (state, category, last 3 months).
`installment_pct` must be computed from the raw `payments` table separately (before aggregation):
```python
installment_pct = (
    payments[payments["order_id"].isin(relevant_order_ids)]
    .query("payment_type == 'credit_card'")
    ["payment_installments"]
    .ge(6)
    .mean()
)
```

**Decision:** Fix join chain in spec. User confirmed.

### Finding 3 — [P2] (confidence: 7/10) `seller_state` Filter Is a Proxy, Not True Supply Capacity

`SupplyQualityAgent` filters `seller_state == state` to count sellers for that state. In Olist, 60% of sellers are in SP. For RJ, this counts sellers physically IN RJ — but most RJ demand is served by SP sellers. So `seller_count` for RJ will be systematically lower than actual supply capacity.

**This is intentional** per the plan's framing ("recruit LOCAL sellers"). But the code comment must say so:
```python
# seller_state == state: counts sellers located in this state (proxy for local supply).
# Does not count cross-state sellers serving this state. Intentional: we want
# to identify where Olist should recruit LOCAL sellers to reduce SP dependency.
```

No spec change needed — just a mandatory comment in the implementation.

### Finding 5 — [P1] (confidence: 9/10) Reviewer Concern #3 Still Open — `delivery_variance` NaN Guard

The plan notes this in Reviewer Concerns but does NOT include the guard in the spec code. This is a production bug: `numpy.std()` returns NaN for 1 data point, not 0.0. The acceptance criterion says "No NaN in any field."

**Fix — must be explicit in spec:**
```python
delivered_days = df_pair["delivery_days"].dropna()
delivery_variance = float(delivered_days.std()) if len(delivered_days) >= 2 else 0.0
```

Note: `len(delivered_days) >= 2` not `> 0` — std() on 1 point gives NaN. This guard must be written exactly this way.

### Finding 6 — [P2] (confidence: 8/10) `seller_concentration` Undefined When seller_count == 0

Plan says `top_seller_id = ""` when no sellers. But `seller_concentration = revenue of top_seller / max(total, 0.01)` — if seller_count == 0, "revenue of top_seller" is undefined.

**Fix — explicit guard in spec:**
```python
if seller_count == 0:
    top_seller_id = ""
    seller_concentration = 0.0   # explicit, not pandas-derived
else:
    top_seller_id = df_pair.groupby("seller_id")["price"].sum().idxmax()
    seller_concentration = (
        df_pair[df_pair["seller_id"] == top_seller_id]["price"].sum()
        / max(df_pair["price"].sum(), 0.01)
    )
```
