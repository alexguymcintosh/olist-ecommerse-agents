#### Python Interface

```python
import sqlite3
from pathlib import Path
from typing import Any
import pandas as pd


class Memory:
    DEFAULT_DB = Path(__file__).resolve().parents[1] / "memory.db"

    def __init__(self, db_path: str | Path | None = None) -> None:
        self._path = Path(db_path) if db_path else self.DEFAULT_DB
        self._create_table_if_not_exists()

    def write_row(self, state: str, category: str, month: str, **cols: Any) -> None:
        """Upsert: INSERT OR REPLACE INTO agent_memory. Only updates provided columns."""
        # Read existing row first (preserve other agents' columns)
        existing = self.read_row(state, category, month) or {}
        merged = {**existing, **cols, "state": state, "category": category, "month": month}
        columns = ", ".join(merged.keys())
        placeholders = ", ".join(["?"] * len(merged))
        with sqlite3.connect(self._path) as conn:
            conn.execute(
                f"INSERT OR REPLACE INTO agent_memory ({columns}) VALUES ({placeholders})",
                list(merged.values()),
            )

    def read_row(self, state: str, category: str, month: str) -> dict | None:
        with sqlite3.connect(self._path) as conn:
            conn.row_factory = sqlite3.Row
            cur = conn.execute(
                "SELECT * FROM agent_memory WHERE state=? AND category=? AND month=?",
                (state, category, month),
            )
            row = cur.fetchone()
            return dict(row) if row else None

    def read_prev_row(self, state: str, category: str, month: str) -> dict | None:
        """Return memory row for the month immediately before `month`."""
        import pandas as pd
        prev = str(pd.Period(month, freq="M") - 1)
        return self.read_row(state, category, prev)

    def read_all(self) -> pd.DataFrame:
        with sqlite3.connect(self._path) as conn:
            return pd.read_sql("SELECT * FROM agent_memory ORDER BY state, category, month", conn)

    def _create_table_if_not_exists(self) -> None:
        with sqlite3.connect(self._path) as conn:
            conn.execute("""CREATE TABLE IF NOT EXISTS agent_memory (
                state TEXT NOT NULL, category TEXT NOT NULL, month TEXT NOT NULL,
                geo_predicted_growth REAL, geo_actual_growth REAL,
                geo_directional_accuracy INTEGER, geo_confidence TEXT,
                geo_confidence_score REAL, geo_reasoning TEXT,
                sq_seller_count INTEGER, sq_avg_review REAL,
                sq_avg_delivery_days REAL, sq_churn_risk TEXT,
                sq_top_seller_id TEXT, sq_reasoning TEXT,
                cr_avg_spend REAL, cr_order_volume_trend REAL,
                cr_top_payment_type TEXT, cr_high_value_customer_count INTEGER,
                cr_repeat_rate REAL, cr_reasoning TEXT,
                log_avg_delivery_days REAL, log_pct_on_time REAL,
                log_freight_ratio REAL, log_fastest_seller_state TEXT,
                log_delivery_variance REAL, log_reasoning TEXT,
                conn_decision TEXT, conn_confidence TEXT,
                conn_reasoning TEXT, conn_actual_outcome TEXT,
                conn_most_predictive_agent TEXT,
                PRIMARY KEY (state, category, month)
            )""")
```

**Acceptance criteria:**
- `write_row()` is non-destructive: only updates provided keyword columns, preserves existing columns
- `read_prev_row()` returns None for the first iteration (no previous month exists) without error
- `read_all()` returns a DataFrame with all 32 schema columns (NaN for unwritten cells) — count: 3 PK + 6 geo + 6 sq + 6 cr + 6 log (incl log_delivery_variance) + 5 conn = 32
- Table auto-creates on first instantiation — no migration needed
- All 5 agents pass different column subsets to `write_row()` — they never overwrite each other

**Error handling:**
- SQLite disk-full or locked: let exception propagate — fatal, not swallowable
- `read_row()` on missing row: return None (callers must handle None)

**Explicitly deferred:** Multi-process safety, WAL mode, connection pooling, memory inspection CLI.

### 4. agents/supply_quality/supply_quality_agent.py (Agent 2)

**Purpose:** Seller health signal. For each (state, category, month) in the training window, assess whether existing sellers can fulfil predicted demand.

**Input tables:** `sellers + order_items + reviews + orders + products + categories`

**Join:**
```python
order_items
  .merge(orders, on="order_id", how="inner")
  .merge(sellers, on="seller_id", how="left")
  .merge(reviews, on="order_id", how="left")
  .merge(products, on="product_id", how="left")
  .merge(categories, on="product_category_name", how="left")
```

Note: `products + categories` are required to filter by `product_category_name_english`.

**Pandas computes (ALL metrics before any LLM call):**

| Metric | Computation |
|--------|-------------|
| `seller_count` | `df[(seller_state==state) & (category==cat)]["seller_id"].nunique()` |
| `avg_review_score` | weighted mean of `review_score` over last 3 months of training window; sentinel 3.0 if < 1 review |
| `avg_delivery_days` | mean of `(order_delivered_customer_date - order_purchase_timestamp).dt.days` where `order_status == "delivered"` and seller in (state, category) |
| `churn_rate` | `count(seller_ids with ≥1 order in months[end-7:end-5] AND 0 orders in month[end-1]) / max(count(all seller_ids with any order in training window), 1)` — 0.0 if training window < 7 months |
| `churn_risk` | "HIGH" if churn_rate > 0.3; "MEDIUM" if 0.1–0.3; "LOW" if < 0.1 |
| `top_seller_id` | seller_id with most order_items in (state, category); "" if 0 sellers |
| `seller_concentration` | `0.0` if seller_count == 0; else `revenue of top_seller / max(total category revenue in state, 0.01)`. Explicit guard required — "revenue of top_seller" is undefined when there are no sellers. |

**LLM call (one per state×category):**
```
Prompt: "Seller metrics for {category} in {state} state, training to {month}:
- seller_count: {N}
- avg_review_score: {X}/5.0
- avg_delivery_days: {D}
- churn_risk: {HIGH/MEDIUM/LOW} (churn_rate={R:.1%})
- top_seller_id: {id}, seller_concentration: {C:.1%}

Assess supply confidence and return JSON:
{\"supply_confidence\": \"STRONG|ADEQUATE|WEAK\", \"reasoning\": \"...\", \"risk_flags\": [...]}"
```

**Confidence mapping for connector composite score:**
```python
SUPPLY_SCORE = {"STRONG": 1.0, "ADEQUATE": 0.5, "WEAK": 0.0}
```

**Interface:**
```python
class SupplyQualityAgent:
    AGENT_NAME = "supply_quality"

    def __init__(
        self,
        data: dict[str, pd.DataFrame] | None = None,
        model: str = RND_MODEL,
    ) -> None: ...

    def run(
        self,
        training_df: pd.DataFrame,     # pre-joined full dataset, filtered to training window
        states: list[str],
        categories: list[str],
        month: str,                    # prediction month as "YYYY-MM" string
        prev_memory: dict[tuple[str, str], dict] | None = None,  # keyed by (state, category) → dict of sq_* columns from prev month; None on first iteration
    ) -> list[SupplyQualityOutput]: ...
```

**Memory write (called by walk_forward_full.py, not by agent):**
```python
memory.write_row(state, category, month,
    sq_seller_count=output["seller_count"],
    sq_avg_review=output["avg_review_score"],
    sq_avg_delivery_days=output["avg_delivery_days"],
    sq_churn_risk=output["churn_risk"],

```

**Hard flags:**
- `seller_count == 0` → always add `"critical_seller_gap"` to risk_flags regardless of LLM output
- `churn_rate > 0.5` → add `"high_seller_churn"` to risk_flags

**Connector tuning target:** If geo predicts HIGH growth AND supply_confidence == "WEAK" → connector flags "high_risk_opportunity" in ConnectorDecision.risk_flags.

**Sparse pair handling:** State×category pairs with order_count < MIN_MONTHLY_ORDERS in the most recent training month get `risk_flags: ["sparse_data"]` and LLM assessment is skipped — a sentinel output is returned instead. The agent always returns exactly 25 outputs regardless.

**Acceptance criteria:**
- Returns exactly `len(states) × len(categories)` = 25 SupplyQualityOutput objects
- Low-volume pairs (< MIN_MONTHLY_ORDERS) return `risk_flags: ["sparse_data"]` with sentinel values, not an error
- All 25 objects have valid `supply_confidence` in {STRONG, ADEQUATE, WEAK}
- seller_count == 0 always produces "critical_seller_gap" risk_flag
- LLM parse failure returns fallback output with `risk_flags: ["agent_failed"]` (never raises)
- Tests in `tests/supply_quality/test_supply_quality_agent.py`

**Explicitly deferred:** Seller-level time series, weighted review decay, multi-category concentration.

### 5. agents/customer_ready/customer_ready_agent.py (Agent 3)

**Purpose:** Buyer signal. For each (state, category, month), assess whether customers in that state are ready to spend in that category next month.

**Input tables:** `orders + customers + order_items + products + categories + payments`

**Join:**
```python
# ⚠️ CRITICAL: payments has N rows per installment (payment_sequential).
# Pre-aggregate to order level BEFORE joining — otherwise ALL metrics are corrupted.
payments_agg = (
    payments
    .groupby("order_id")["payment_value"]
    .sum()
    .reset_index()
    .rename(columns={"payment_value": "total_payment_value"})
)

orders
  .merge(customers, on="customer_id", how="inner")
  .merge(order_items, on="order_id", how="inner")
  .merge(products, on="product_id", how="left")
  .merge(categories, on="product_category_name", how="left")
  .merge(payments_agg, on="order_id", how="left")   # aggregated — safe
```

`installment_pct` must be computed from raw `payments` BEFORE aggregation (separate pass):
```python
installment_pct = (
    payments[payments["order_id"].isin(relevant_order_ids)]
    .query("payment_type == 'credit_card'")
    ["payment_installments"]
    .ge(6)
    .mean()
)
```

**CRITICAL join note:** After merging `payments_agg`, `main_df` has 1 row per order_item (because order_items has 1 row per item in each order). A 3-item order has 3 rows each carrying the same `total_payment_value`. Before computing any money-based metrics, collapse to one row per order:

```python
per_order = main_df.drop_duplicates(subset="order_id")[
    ["order_id", "customer_unique_id", "customer_state", "product_category_name_english",
     "total_payment_value", "month"]
]
# repeat_rate and order counts still use main_df (order_item level for order_id counts)
# but money metrics (avg_spend, high_value_customer_count) use per_order
```

**Pandas computes (ALL metrics before any LLM call):**

| Metric | Computation |
|--------|-------------|
| `avg_spend` | `per_order["total_payment_value"].mean()` — uses order-level df (not order_item rows) |
| `order_volume_trend` | MoM order count: `main_df["order_id"].nunique()` per month — count distinct orders, not rows |
| `top_payment_type` | mode of `payment_type` from raw payments for this state; default "credit_card" |
| `high_value_customer_count` | `customer_unique_id`s whose avg `total_payment_value` > `per_order["total_payment_value"].median()` — uses order-level df |
| `repeat_rate` | `customer_unique_id`s with > 1 distinct `order_id` in (state, category) / total unique customers |
| `installment_pct` | computed from raw payments (separate pass, before aggregation); `.drop_duplicates("order_id")` on credit_card rows, then `.ge(6).mean()` |

**Temporal note:** "last 3 months" = months [-3, -2, -1] relative to training window max month.

**LLM call (one per state×category):**
```
Prompt: "Customer metrics for {category} in {state} state, training to {month}:
- avg_spend: R${avg_spend:.2f}
- order_volume_trend: {order_volume_trend:+.1%} MoM
- top_payment_type: {top_payment_type}
- high_value_customer_count: {N}
- repeat_rate: {repeat_rate:.1%}
- installment_pct: {installment_pct:.1%}

Assess customer readiness and return JSON:
{\"readiness\": \"HIGH|MEDIUM|LOW\", \"reasoning\": \"...\", \"risk_flags\": [...]}"
```

**Confidence mapping for connector composite score:**
```python
READINESS_SCORE = {"HIGH": 1.0, "MEDIUM": 0.5, "LOW": 0.0}
```

**Known data fact:** Repeat rate is 0% system-wide. Per-state-category rate will also be ~0. Agent should report this truthfully — it is a signal, not a bug.

**Interface:** Identical structure to SupplyQualityAgent — replace type names.

**Memory write:**
```python
memory.write_row(state, category, month,
    cr_avg_spend=output["avg_spend"],
    cr_order_volume_trend=output["order_volume_trend"],
    cr_top_payment_type=output["top_payment_type"],
    cr_high_value_customer_count=output["high_value_customer_count"],
    cr_repeat_rate=output["repeat_rate"],
    cr_reasoning=output["reasoning"],
)
```

**Connector tuning target:** If geo predicts growth AND readiness == "LOW" → connector recommends marketing spend over seller recruitment.

**Acceptance criteria:**
- 25 outputs, all with valid `readiness` in {HIGH, MEDIUM, LOW}
- repeat_rate = 0.0 is valid output (not an error condition)
- `installment_pct` only computed on credit_card orders (boleto/voucher cannot have installments)
- LLM parse failure → fallback with `risk_flags: ["agent_failed"]`

**Explicitly deferred:** Category-level spend segmentation, customer lifetime value, cohort analysis.

| `pct_on_time` | fraction where `order_delivered_customer_date <= order_estimated_delivery_date`; 0.0 if no delivered orders |
| `freight_ratio` | `mean(freight_value) / max(mean(price), 0.01)` in (state, category) |
| `fastest_seller_state` | `seller_state` with lowest mean delivery days to `customer_state==state`; "" only when there are zero delivered orders (data unavailable) |
| `delivery_variance` | `float(delivered_days.std()) if len(delivered_days) >= 2 else 0.0` — MUST use `>= 2` guard; `numpy.std()` returns NaN (not 0.0) for exactly 1 data point |
| `cross_state_dependency` | fraction of orders where `seller_state != customer_state` |

**State×category scoping:** `avg_delivery_days` and `pct_on_time` scope to the customer's state (demand side). `fastest_seller_state` looks at all seller states serving that customer state.

**LLM call (one per state×category):**
```
Prompt: "Logistics metrics for {category} → {state} customers, training to {month}:
- avg_delivery_days: {D:.1f}
- pct_on_time: {pct_on_time:.1%}
- freight_ratio: {freight_ratio:.2f}
- fastest_seller_state: {fastest_seller_state}
- delivery_variance: {delivery_variance:.1f} days std dev
- cross_state_dependency: {cross_state_dependency:.1%}

Assess logistics feasibility and return JSON:
{\"feasibility\": \"STRONG|ADEQUATE|WEAK\", \"reasoning\": \"...\", \"risk_flags\": [...]}"
```

**Confidence mapping for connector composite score:**
```python
LOGISTICS_SCORE = {"STRONG": 1.0, "ADEQUATE": 0.5, "WEAK": 0.0}
```

**Memory write:**
```python
memory.write_row(state, category, month,
    log_avg_delivery_days=output["avg_delivery_days"],
    log_pct_on_time=output["pct_on_time"],
    log_freight_ratio=output["freight_ratio"],
    log_fastest_seller_state=output["fastest_seller_state"],
    log_delivery_variance=output["delivery_variance"],   # persisted — useful for analysis
    log_reasoning=output["reasoning"],
)
```

**Connector tuning target:** If geo predicts RJ growth AND cross_state_dependency > 0.9 → connector recommends recruiting local RJ sellers not increasing SP inventory.

**GeographicOutput dependency:** ConnectorAgent imports `GeographicOutput` from `utils.schema_geographic`. It assumes these fields exist on each prediction object: `state`, `category`, `predicted_growth_pct`, `confidence`, `confidence_score`. If schema_geographic.py changes these names, connector breaks. Do not rename these fields without updating connector.

**Acceptance criteria:**
- 25 outputs, all with valid `feasibility` in {STRONG, ADEQUATE, WEAK}
- No NaN in any field — all have explicit guards (filter delivered, max(price, 0.01), default "" for fastest_seller_state)
- `delivery_variance` == 0.0 for pairs with < 2 delivered orders (not NaN)
- LLM parse failure → fallback with `risk_flags: ["agent_failed"]`

**Explicitly deferred:** Route-level analysis, carrier performance, freight optimisation.

### 7. agents/connector/connector_agent.py (Agent 5)

**Purpose:** Decision maker. Reads all 4 agent outputs for the same (state, category, month), computes composite score, makes one ranked decision per pair, stores reasoning. Learns from previous month's outcome.

**Does NOT subclass any base class. Standalone interface.**

**Interface:**
```python
class ConnectorAgent:
    AGENT_NAME = "connector"

    def __init__(
        self,
        memory: Memory,
        model: str = RND_MODEL,
    ) -> None: ...

    def run(
        self,
        month: str,
        geographic_outputs: list[GeographicOutput],
        supply_outputs: list[SupplyQualityOutput],
        customer_outputs: list[CustomerReadinessOutput],
        logistics_outputs: list[LogisticsOutput],
        prev_month: str | None = None,
    ) -> ConnectorOutput: ...
```

**Composite score (pandas, not LLM):**
```python
SUPPLY_SCORE  = {"STRONG": 1.0, "ADEQUATE": 0.5, "WEAK": 0.0}
READINESS_SCORE = {"HIGH": 1.0, "MEDIUM": 0.5, "LOW": 0.0}
LOGISTICS_SCORE = {"STRONG": 1.0, "ADEQUATE": 0.5, "WEAK": 0.0}

def _composite_score(geo, supply, customer, logistics) -> float:
    geo_signal = geo["confidence_score"] * 0.35
    supply_signal = SUPPLY_SCORE.get(supply["supply_confidence"], 0.0) * 0.25
    customer_signal = READINESS_SCORE.get(customer["readiness"], 0.0) * 0.20
    logistics_signal = LOGISTICS_SCORE.get(logistics["feasibility"], 0.0) * 0.20
    return (geo_signal + supply_signal + customer_signal + logistics_signal) * geo["predicted_growth_pct"]
    # NOTE: if predicted_growth_pct == 0.0 or negative, composite_score is 0.0 or negative.
    # Decisions with composite_score <= 0 are labelled "no_action" with urgency "LOW".
    # This is intentional: growth is the demand signal; zero/negative growth means no opportunity.
```

**LLM call (one per state×category — 25 total):**

The connector reads previous month's outcome from memory before building the prompt:

```python
prev_row = memory.read_row(state, category, prev_month) if prev_month else None
# NOTE: read_row(prev_month) not read_prev_row(prev_month) — prev_month is already the prev month string
prev_outcome_text = f"Last month outcome: {prev_row['conn_actual_outcome']}" if prev_row and prev_row.get('conn_actual_outcome') else "First iteration — no prior outcome."
```

```
Prompt: "Multi-signal analysis for {category} in {state}, month {month}:

COMPOSITE SCORE: {composite_score:.2f}
GEO: predicted_growth={predicted_growth_pct:+.1%}, confidence={confidence} ({confidence_score:.2f})
SUPPLY: {supply_confidence} — {sq_reasoning or "Supply metrics unavailable (agent failed)"}
CUSTOMER: {readiness} — {cr_reasoning or "Customer metrics unavailable (agent failed)"}
LOGISTICS: {feasibility} — {log_reasoning or "Logistics metrics unavailable (agent failed)"}

{prev_outcome_text}

What is the single best action Olist should take? Challenge your own recommendation — what could go wrong?

Return JSON:
{
  "decision": "...",
  "confidence": "HIGH|MEDIUM|LOW",
  "urgency": "HIGH|MEDIUM|LOW",
  "reasoning": "...",
  "challenge": "...",
  "most_predictive_agent": "geographic|supply_quality|customer_readiness|logistics"
}"
```

**Optional follow-up loop (one follow-up per full run, not per pair):**

After all 25 initial decisions are made, the connector identifies the top-ranked opportunity and optionally sends a follow-up LLM question to clarify one signal before finalising the final briefing:

```python
# One follow-up, triggered only if composite_score variance is high
# (highest score is > 2× second-highest — strong signal demands clarification)
# Guard: if fewer than 2 decisions, follow_up_used=False, no exception
top = sorted_decisions[0]
if len(sorted_decisions) >= 2:
    runner_up = sorted_decisions[1]
    should_follow_up = top.composite_score > 2 * max(runner_up.composite_score, 0.001)
else:
    should_follow_up = False
if should_follow_up:
    # Follow-up: ONE LLM call, ONE pair (top-ranked only), plain text response (no JSON)
    # Context sent: only top pair's 4 agent signals + initial decision + challenge text
    # Not sent: other 24 pairs, full DataFrames, raw data
    follow_up_q = f"Why is {top.state}×{top.category} the top opportunity? What would change this assessment?"
    follow_up_context = (
        f"Initial decision: {top.decision}\nInitial challenge: {top.challenge}\n"
        f"GEO: growth={geo_for_pair['predicted_growth_pct']:+.1%}, confidence={geo_for_pair['confidence']}\n"
        f"SUPPLY: {sq_for_pair['supply_confidence']} — {sq_for_pair['reasoning']}\n"
        f"CUSTOMER: {cr_for_pair['readiness']} — {cr_for_pair['reasoning']}\n"
        f"LOGISTICS: {log_for_pair['feasibility']} — {log_for_pair['reasoning']}"
    )
    follow_up_response = query_llm(build_analyst_prompt(follow_up_context, follow_up_q), model=self.model)
    ...
```

The follow-up is ONE additional LLM call for ONE pair (the top-ranked opportunity), not a re-run of the full 25. This is the "back and forth loop" from CLAUDE.md.

**Memory write:**
```python
# Per decision
memory.write_row(state, category, month,
    conn_decision=decision["decision"],
    conn_confidence=decision["confidence"],
    conn_reasoning=decision["reasoning"],
    conn_most_predictive_agent=decision["most_predictive_agent"],
)
# conn_actual_outcome is written by walk_forward_full.py during validation (next iteration)
```

**Output — ConnectorOutput:**
- `decisions` sorted by `composite_score` descending (rank 1 = highest priority)
- `briefing` — Rich terminal narrative summarising top 3 decisions
- `follow_up_used`, `follow_up_agent`, `follow_up_question`, `follow_up_response`

**Acceptance criteria:**
- Returns exactly 25 ConnectorDecision objects, sorted by composite_score desc
- LLM parse failure on a decision → fallback with `risk_flags: ["connector_failed"]`, composite_score preserved
- `prev_month=None` (first iteration) runs without error
- If any domain agent output has `risk_flags: ["agent_failed"]`, the connector notes it in decision.risk_flags but still makes a decision from available signals
- `follow_up_used=False` if composite_score variance is low (no ambiguity)
- Tests in `tests/connector/test_connector_agent.py`

**Explicitly deferred:** Auto-adjusting composite weights based on most_predictive_agent history. Weight adjustment tracked in TODOS.md.

### 8. walk_forward_full.py

**Scope:** 5-agent orchestrator. Sequential iterations Sep 2017 → Aug 2018. Calls all 5 agents per iteration. Writes to memory. Validates previous predictions.

**Interface:**
```
Usage: python walk_forward_full.py [--db PATH] [--max-iterations N]

  --db            SQLite memory path (default: memory.db)
  --max-iterations  Override MAX_ITERATIONS from config (default: 13)
```

**Relationship to walk_forward.py:** `walk_forward.py` (geographic-only) needs a minor update to import states/categories from `utils.config` (currently hardcoded). Not "unchanged" — note this in implementation.

**training_df pattern (resolved by eng review):** `training_df` is a base joined df (orders + customers + order_items + products + categories) pre-filtered to the training window. Each agent receives this and adds only its agent-specific tables on top (supply adds sellers+reviews, customer adds payments_agg, logistics adds sellers+customers already present).

```python
# In walk_forward_full.py — build base_df ONCE before the loop
base_df = (
    data["orders"]
    .merge(data["customers"], on="customer_id", how="inner")
    .merge(data["order_items"], on="order_id", how="inner")
    .merge(data["products"], on="product_id", how="left")
    .merge(data["categories"], on="product_category_name", how="left")
)
base_df["month"] = pd.to_datetime(base_df["order_purchase_timestamp"]).dt.to_period("M")
# base_df is reused across all iterations; temporal_window slices it each iteration
```

**prev_memory construction:** Before calling agents 2-4, walk_forward_full.py reads previous month rows from memory and passes them as a dict:
```python
prev_sq = {(s, c): memory.read_prev_row(s, c, month) or {} for s in states for c in categories}
```

**Per-iteration flow:**
```python
prev_prediction_month = None  # carry-forward — None on first iteration
for iteration in range(1, MAX_ITERATIONS + 1):
    training_window, prediction_month = compute_windows(iteration)
    training_df, _ = temporal_window(base_df, start_month, end_month)
    # NOTE: temporal_window() returns (df, month_count) tuple — unpack with _

    # 1. Run agents (sequential — no async in MVP)
    # ⚠️ No checkpoint/resume — mid-run LLM failure aborts all iterations (TODOS.md #1)
    geo_outputs = GeographicAgent(data, model).run(training_df, ...)
    prev_sq = {(s, c): memory.read_prev_row(s, c, month) or {} for s in states for c in categories}
    sq_outputs  = SupplyQualityAgent(data, model).run(training_df, states, categories, month, prev_memory=prev_sq)
    prev_cr = {(s, c): memory.read_prev_row(s, c, month) or {} for s in states for c in categories}
    cr_outputs  = CustomerReadinessAgent(data, model).run(training_df, states, categories, month, prev_memory=prev_cr)
    prev_log = {(s, c): memory.read_prev_row(s, c, month) or {} for s in states for c in categories}
    log_outputs = LogisticsAgent(data, model).run(training_df, states, categories, month, prev_memory=prev_log)

    # 2. Write domain signals to memory
    for output in geo_outputs["predictions"]:
        memory.write_row(output["state"], output["category"], month, geo_predicted_growth=..., ...)
    for sq in sq_outputs:
        memory.write_row(sq["state"], sq["category"], month, sq_seller_count=..., ...)
    # same for cr, log

    # 3. Run connector
    conn_output = ConnectorAgent(memory, model).run(
        month=month,
        geographic_outputs=geo_outputs["predictions"],
        supply_outputs=sq_outputs,
        customer_outputs=cr_outputs,
        logistics_outputs=log_outputs,
        prev_month=prev_prediction_month,   # None on iteration 1, prev month string thereafter
    )
    for decision in conn_output["decisions"]:
        memory.write_row(decision["state"], decision["category"], month, conn_decision=..., ...)

    # 4. Validate previous iteration (write conn_actual_outcome)
    if iteration > 1:
        _validate_prev_iteration(joined_df, memory, prev_prediction_month)

    prev_prediction_month = prediction_month   # carry forward for next iteration
    print(f"Iteration {iteration}/{MAX_ITERATIONS} complete — {prediction_month}")
```

**Validation:** `conn_actual_outcome` = "correct" if actual growth direction matches connector's decision direction, "incorrect" otherwise. Written to memory for the previous month.

## Error & Rescue Map

```
CODEPATH                            | WHAT CAN GO WRONG           | RESCUED? | USER SEES
------------------------------------|-----------------------------|---------|-----------
SupplyQualityAgent._assess_supply() | LLM malformed JSON          | YES      | fallback output, risk_flags: ["agent_failed"]
CustomerReadinessAgent._assess()    | LLM malformed JSON          | YES      | fallback output, risk_flags: ["agent_failed"]
LogisticsAgent._assess()            | LLM malformed JSON          | YES      | fallback output, risk_flags: ["agent_failed"]
ConnectorAgent._decide()            | LLM malformed JSON          | YES      | fallback decision, risk_flags: ["connector_failed"]
ConnectorAgent._decide()            | Missing agent output pair   | YES      | skip pair, log warning
Memory.write_row()                  | SQLite disk full / locked   | NO       | exception propagates (fatal — expected)
Memory.read_row()                   | Missing row (first iter)    | YES      | returns None, callers handle
_compute_churn_rate()               | Division by zero (0 sellers)| YES      | guard: max(total_sellers, 1)
_compute_freight_ratio()            | avg price = 0               | YES      | guard: max(avg_price, 0.01)
_compute_delivery_days()            | NaT delivery dates          | YES      | filter: order_status == "delivered"
_compute_delivery_variance()        | < 2 delivered orders        | YES      | return 0.0 (not NaN)
_compute_avg_review_score()         | No reviews for pair         | YES      | return sentinel 3.0
```

**CRITICAL: All LLM parse failures use the brace-window extraction pattern from geographic_agent.py:**
```python
raw[raw.find("{"):raw.rfind("}") + 1]
```
This tolerates prose prefix before JSON (known pitfall: `build_analyst_prompt()` system message says "Use bullet points" which conflicts with JSON-only output).
